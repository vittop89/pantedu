### Fix nginx blocklist su VPS

`/etc/nginx/sites-enabled/beta.pantedu.eu` aveva regex blocklist:
```
location ~ ^/(storage|database|tools|tests|wiki|docs|app|bootstrap|composer\.|vendor)/ { deny all; }
```
La parola `vendor` era safety per nascondere composer vendor folder, ma quel
folder è FUORI dal webroot (`/var/www/pantedu/vendor`, webroot è `public/`),
quindi era ridondante. Rimossa per permettere `/vendor/quill/*` serving.

Modifica applicata MANUALMENTE su VPS in `sites-enabled/beta.pantedu.eu`
(file standalone, NON symlink). Backup salvato in `.bak-q4`.

- `beta.pantedu.eu` (file standalone — quello servito)
- `beta.pantedu.eu.bak.pantedu` → symlink a `sites-available/beta.pantedu.eu`

Causa "conflicting server name" warning su `nginx -t`. NON bloccante ma da
cleanup futuro. Inoltre i due file sono DESINCRONIZZATI:

## 2026-05-19 — Phase 25.K — VPS hardening Tier 1-4 + backup encrypted offsite B2

Completata l'intera pipeline di hardening VPS (`tools/dev/hardening/01-12-*.sh`)
e backup offsite cifrato verso Backblaze B2.

### Stack hardening attivo

- **Tier 1 (system)**: SSH `AllowUsers root` + `MaxAuthTries 3` + `ClientAliveInterval`,
  `AllowTcpForwarding local` (permette `-L` per Grafana tunnel ma blocca `-R`),
  sysctl (SYN cookies, ASLR, `kptr_restrict=2`), unattended-upgrades auto-reboot.
- **Tier 2 (WAF)**: fail2ban jail custom con PHP exporter, AIDE integrity check,
  Lynis audit, CT log monitor (`crt.sh`).
- **Tier 3 (defense in depth)**: ModSecurity v3 + OWASP CRS (DetectionOnly),
  Suricata NIDS con ET Open rules (~40k), Loki + Promtail + Grafana stack.
- **Tier 4 (data layer)**: MariaDB `server_audit` plugin attivo, Grafana bound
  127.0.0.1 (admin access solo via SSH tunnel).

### Backup encrypted offsite (Phase 25.K.4 + .11 + .12)

- `tools/backup/encrypted_backup.sh` — script orchestratore:
  1. `mysqldump --user=root` (auth via `/etc/pantedu/backup.env`)
  2. `tar storage/` con `zstd -3 -T0` (exclude cache/sessions/logs)
  3. `tar config + secrets` (.env, .env.local, app/Config/)
  4. `gpg --symmetric --cipher-algo AES256` (passphrase 32-char random)
  5. Upload Backblaze B2 via rclone (backend **B2 native**, NON S3-compat
     per via di limitazioni "not entitled" su PUT in modalità S3).
- **Tiered retention B2**: daily ≤ 7gg → keep all, 8-30gg → 1/settimana ISO,
  31-365gg → 1/mese, > 365gg → 1/anno. Implementato in awk via `mktime` +
  `strftime("%G-%V")`. Costo stimato: €0-3/anno per bucket < 1TB.
- **TMPDIR** spostato da `/tmp` (tmpfs 1.9G — esaurito al primo run) a
  `/var/backups/pantedu/.tmp` (su `/dev/sda1` 75G).
- Systemd timer attivo: 02:30 ogni notte (low traffic).

### Test restore validato

`pantedu-backup-20260519_2055.tar.gpg` (2.6G):
- SHA256 verify OK
- GPG decrypt OK → bundle tar 2.7G
- Bundle contiene 3 file: `db_*.sql.gz` (52 CREATE TABLE, 6MB SQL),
  `storage_*.tar.zst` (8692 file), `config_*.tar.gz` (.env, .env.local +
  16 file app/Config/).

### Decisioni architetturali emerse

- **PHP-FPM systemd sandbox skip**: tentato `ProtectProc`, `NoNewPrivileges`,
  `ProtectHome` ma incompatibili con webhook deploy (sudo + ssh known_hosts).
  Rollback completo, documentato come architectural mismatch in
  [`docs/todo/waf_security_prompt.md`](../../docs/todo/waf_security_prompt.md).
- **rclone B2 native ≠ S3-compat**: chiave R+W validata via `b2_authorize_account`
  ma `s3-compat` endpoint restituiva 403 "not entitled" anche con region set.
  Switch a backend `type = b2` risolutivo.
- **MariaDB auth Trixie**: `/root/.my.cnf` necessario per root password auth
  (default password rifiutata, plugin = mysql_native_password); user grant
  `'pantedu_app'@'127.0.0.1'` ≠ `'pantedu_app'@'localhost'`.

### File coinvolti

`tools/dev/hardening/01-12-*.sh`, `tools/backup/encrypted_backup.sh`,
`/etc/pantedu/backup.env` (VPS, 600 root:root, contiene
`BACKUP_GPG_PASSPHRASE` + `DB_ROOT_PASSWORD` + `B2_KEY_ID` + `B2_APP_KEY`),

## 2026-05-13 — G27.tikz.hoist — Preamble macro hoisting per template TikZ con \newcommand

  popolato da PhpContentParser ma mai consumato dal pipeline TeX).
- **`TexBuilder::build()`**: chiama `Sanitizer::resetHoistedPreamble()` prima

## 2026-05-12 — G23.fix7 — Enter su LI vuoto nested distrugge nested struttura

### Bug

Su una lista nested:
```
1. u
   a. uu
      i. uuu
      ii.    ← LI vuoto qui
```

Premendo Enter su `ii.` (LI vuoto al livello 3), la struttura si distorce:
- Si crea un nuovo "2." al top level ✓ (atteso)
- MA i marker nested cambiano: `a. uu` → `1. uu`, `i. uuu` → `a. uuu`

### Root cause

`_outdentListItem(li, field, removeIfEmpty=true)` SEMPRE creava un
`<div><br></div>` come container per il nuovo blocco — anche quando il
li vuoto era nidificato. Quando inserito dentro un `<ol>` (sibling di
LI), il `<div>` produce HTML invalido. Il browser corregge in modo
imprevedibile, ma il CSS selector `> li > ol` non matcha più →
list-style-type cascade rotta → markers cambiano livello.

### Fix

Riscritto il branch `removeIfEmpty` per distinguere TOP-LEVEL vs NESTED:

**Nested** (parent OL è dentro un LI):
- Crea nuovo `<li>` vuoto come sibling di `grandLi`
- Caret nel nuovo LI
- HTML valido → CSS cascade preservata → markers corretti

**Top-level** (parent OL è figlio diretto del field):
- Crea `<div><br></div>` come sibling di parentList (FUORI dall'OL)
- Caret nel div
- Esce dalla lista

```js
if (grandLi && grandLi.tagName === "LI") {
    const newLi = document.createElement("li");
    grandList.insertBefore(newLi, grandLi.nextSibling);
    _placeCaretAtStart(newLi);
} else {
    const newBlock = document.createElement("div");
    newBlock.appendChild(document.createElement("br"));
    parentList.parentNode.insertBefore(newBlock, parentList.nextSibling);
    _placeCaretAtStart(newBlock);
}
```

### Comportamento atteso

Su `1.u/a.uu/i.uuu/ii.(empty)`:
- Enter 1: outdent → `1.u/a.uu/i.uuu/b.(empty caret)` (livello 1)
- Enter 2: outdent → `1.u/a.uu/i.uuu` + `2.(empty caret)` (top level)

I marker nested PRESERVATI ad ogni Enter.

## 2026-05-10 — G22.S15.bis Fase 6 — Resize sync bidirezionale editor ⇄ preview

Nuovo modulo `js/modules/editor/resize-sync.js` (`installResizeSync(ta, pv)`)
collegato da `bindPreview()` in `checkin-handlers.js`. Quando l'utente
trascina il corner-resize CSS della textarea (`.fm-editor-field`), il
preview affianco (`.fm-editor-preview`) ne segue l'altezza esatta
(`ResizeObserver` su ta → `pv.style.height`). Quando il contenuto del
preview cresce naturalmente (es. SVG TikZ alto), la textarea adatta il
proprio `minHeight` (`MutationObserver` sul subtree pv + misura altezza
naturale temporaneamente azzerando `pv.style.height`).

**Anti-loop**: A usa ResizeObserver, B usa MutationObserver — i due canali
non si scatenano a vicenda (B reagisce a mutazioni DOM, non dimensioni;
A modifica solo height del pv senza mutare DOM). Cleanup automatico via
MutationObserver sul parent del ta che disconnette tutti gli observer
quando ta esce dal DOM (chiusura editor) → no memory leak.

Rimossa la sync legacy in `updatePreview()` (linee 1915-1925) che
imponeva `ta.minHeight = pv.offsetHeight` post-render: bloccava l'utente
dal poter rimpicciolire la textarea dopo un drag in giù.

Compatibilità futura CodeMirror: il modulo opera su `offsetHeight` e
`scrollHeight`, agnostico al tagName. Quando il textarea sarà sostituito
da un wrapper CM6, basterà passare il wrapper a `installResizeSync`.

## 2026-05-06 — Phase G22.S8 — Stale-job cleanup (chiude trade-off S7)

Chiude il trade-off documentato in S7: se PHP `max_execution_time` killa
una request mentre era in inline-process (trigger-on-request), il job
restava in stato `running` con `started_at` recente; il cron non lo
ripiccava (status != pending/retry); il browser polling continuava a
vederlo `running` indefinitamente → UX bloccata fino al cleanup
manuale di un admin.

### Soluzione

**`Repository::resetStuckRunning(staleSec=300)`** — UPDATE atomic
che setta `status='retry'`, `next_attempt_at=NOW`, `last_error=
'stuck_running_reset_after_<N>s'` per tutti i job con
`status='running'` AND `started_at < NOW - <N>s`.

`attempts` resta INVARIATO (l'incremento e' avvenuto al pick precedente
quando il job e' stato lockato; non penalizziamo l'utente per una
morte di FPM-side).

`next_attempt_at = NOW` permette il pick immediato al prossimo tick
(il backoff non si applica a stuck-recovery, solo ai retry da fail
esplicito).

### Integrazione

Chiamato all'inizio di:
- `Service::processNext()` (FIFO worker) — ogni cron tick libera
  eventuali stuck dei tick precedenti.
- `Service::processJob(jobId)` (trigger-on-request) — defensive:
  se questo specifico job e' stuck, lo libera prima di ri-tentare.
  Cosi' il browser polling che chiama `getJob` su un job
  pre-terminato puo' processarlo subito.

### Smoke test

```bash
# Setup: insert running job old 10 min
INSERT INTO verifica_compile_jobs (..., status='running',
  started_at=DATE_SUB(NOW(), INTERVAL 600 SECOND), attempts=1);

# Trigger
php -r '... resetStuckRunning(300) ...'
# reset count=1

# Verifica
SELECT status, attempts, last_error
# status='retry', attempts=1, last_error='stuck_running_reset_after_300s'
```

### Trade-off rimanenti

- **5 min finestra "limbo"**: un job stuck dopo PHP timeout aspetta
  fino a 5 min prima del reset (al prossimo tick). Acceptable: il
  caso e' raro (compile > 30s e' edge), e l'utente vede comunque
  un retry.
- **Falso positivo possibile**: un compile legittimo > 5 min su
  bundle pesantissimo verrebbe forzato a retry. Mitigation: il
  parametro 300s e' configurabile, alzabile a 600s+ per casi pesanti.

## 2026-05-06 — Phase G22.S7 — Aruba shared friendly: migration web + trigger-on-request

Adattamenti per il vincolo deploy reale: **Aruba shared hosting senza
SSH e con cron min interval 10 minuti**.

### Problema 1 — niente terminale

`php tools/migrate.php` richiede SSH. Su Aruba shared non c'e'. Senza
una via per applicare le migration al deploy, S2/S4.B/S5 non sono
deployabili.

**Soluzione**: endpoint admin via browser.

- `app/Controllers/AdminMigrateController.php` (nuovo)
- `GET  /admin/migrate` — pagina HTML self-contained con stato
  (executed/pending) + bottone "Esegui migration".
- `GET  /admin/migrate/status` — JSON {executed, pending, counts}
  per script automatici.
- `POST /admin/migrate/run` — applica tutte le pending; CSRF +
  super_admin only. Idempotente (Migrator skippa quelle gia'
  registrate). Lock atomico Migrator anti-race.

Il super_admin apre `/admin/migrate` dopo il deploy, vede le pending,
clicca, fatto.

### Problema 2 — cron 10 min troppo lento per UX

Cron tick 10 min su Aruba significa: utente preme GENERA → 8 jobs
in coda → primo pickNext arriva fino a 10 min dopo → UI block 10+min.
Inaccettabile.

**Soluzione**: trigger-on-request inline-processing.

1. **`Repository::pickSpecific(jobId)`** atomic UPDATE: lock di un
   job specifico (non FIFO) se status pending/retry+fuori-backoff.
2. **`Service::processJob(jobId)`** + refactor di `processNext` su
   helper privato `compileAndFinalize($job)` riusabile (FIFO worker
   e specific-trigger condividono il core).
3. **`compileAsync` controller**: dopo `enqueue($jobId)` chiama
   `processJob($jobId)` INLINE nella stessa request HTTP. Compile
   tipico 2-5s, ben dentro PHP `max_execution_time` 60s. Successo
   → return 200 sync con doc.has_pdf=true (UX identica al pre-S5
   sync). Fallimento → return 202 con job_id, cron 10min faro' retry.
4. **`getJob` controller**: stesso trigger se polled mentre status
   resta pending/retry. Defensive secondario per casi edge dove
   l'inline iniziale e' fallito ma i retry sono nel frattempo
   tornati eseguibili.

### Risultato pratico

- **Su XAMPP locale (Windows, niente cron)**: il flusso async funziona
  senza cron. La compile gira inline ad ogni POST `/compile-async`.
- **Su Aruba (cron 10min)**: stesso comportamento UX del flusso sync
  pre-S5 (compile inline, 2-5s wait). Il cron 10min interviene SOLO
  per retry di job veramente falliti (es. VPS down momentaneo). Mai
  il primo tentativo aspetta il cron.

### Vantaggi conservati di S5

- Retry automatico su VPS down (cron ogni 10 min ritenta i 'retry').
- Dedup via payload_hash (ricaricare la pagina e riGENERAre = noop).
- Audit trail completo: ogni compile lascia row in DB (status, error,
  attempts, timing).
- Cache S2 sync hit (path veloce ~10ms invariato).

### File toccati

- `app/Controllers/AdminMigrateController.php` (nuovo, ~150 LOC)
- `routes/web.php` (3 nuove rotte admin)
- `app/Repositories/VerificaCompileJobRepository.php` (`pickSpecific`)
- `app/Services/Verifica/VerificaCompileJobService.php` (`processJob`,
  refactor `processNext` su `compileAndFinalize` privato)
- `app/Controllers/VerificaController.php` (compileAsync inline +

### Trade-off

- **PHP max_execution_time bound**: se Aruba ha `max_execution_time=30`

## 2026-05-05 — Phase G22.S5 — Job queue async per compile PDF

Risolve il bottleneck del flusso sincrono: 8 varianti × 2-5s = 16-40s
di richiesta HTTP bloccata. Su Aruba shared, FPM kill a 30s era rischio
reale per compile pesanti (TikZ + tabularx + griglie).

### Problema

Pre-S5, `POST /api/verifica/{id}/compile` era sincrono:
- Decifra TEX dalla manifest → invoca VPS → riceve PDF → cifra → DB.
- Browser bloccato sul fetch per la durata totale (4-40s nel batch).
- Niente retry su VPS down (errore visibile all'utente, perde il batch).
- Niente dedup: ricaricare la pagina e ricliccare GENERA = ricompila.

### Soluzione — queue async

1. **Migration `033_verifica_compile_jobs.sql`**: tabella nuova
   ```
   verifica_compile_jobs:
     id, doc_id, teacher_id, status (pending|running|done|failed|retry),
     payload_hash (sha dedup), attempts, next_attempt_at,
     engine, passes, last_error,
     created_at, started_at, completed_at
   INDEX (status, id) for FIFO worker pickNext
   INDEX (teacher_id, status, created_at DESC) for status UI
   INDEX (teacher_id, doc_id, payload_hash, status) for dedup
   ```

2. **`VerificaCompileJobRepository`** — CRUD + locking:
   - `enqueue()` idempotente (dedup su payload_hash).
   - `pickNext()` atomic via SELECT + UPDATE rowCount: race-safe per
     multi-worker.
   - `markFailedOrRetry()` exp backoff 1m/5m/15m, max 3 attempts.
   - `purgeOlderThan(days)` cleanup row done/failed > 7 giorni.

3. **`VerificaCompileJobService`** — orchestra:
   - `enqueue(teacherId, docId, engine, passes)` — payload_hash =
     sha256(`tex_sha256|variant|engine|passes`). Ritorna {job_id, dedup}.
   - `processNext()` — worker tick: `pickNext` → `compileBundle` o
     `compile` (fallback) → `attachPdf` → `markDone`. Errori →
     `markFailedOrRetry`.
   - `processBatch(max=5)` — loop per cron tick.

4. **Endpoint nuovi** in `VerificaController`:
   - `POST /api/verifica/{id}/compile-async`:
     - Cache hit S2 → ritorna 200 sync con `compile.cache_hit=true`
     - Cache miss → enqueue job, ritorna 202 `{async:true, job_id}`
     - Latency utente: ~10ms invece di 2-5s.
   - `GET /api/verifica/jobs/{jobId}` — status polling.
     Risposta `{job: {status, attempts, last_error, created_at, ...}}`,
     incluso `doc` se status='done'.

5. **Cron worker `tools/cron/process_compile_jobs.php`**:
   - Schedule consigliato: `* * * * *` (ogni minuto)
   - `processBatch(5)` per tick (allineato a rate limit nginx 20r/min).
   - `purgeOld(7)` post-batch.
   - Output: una riga per job + summary JSON (telemetry).

6. **Frontend `topbar-modern.js::compilePdfForDocs`** — refactor:
   - Pool 4 enqueue concorrenti (era 4 compile concorrenti).
   - Per ogni doc: POST compile-async; se 200 cache_hit → done; se
     202 → polling `/jobs/{id}` con backoff 1s/2s/5s/10s, hard
     timeout 3min.
   - UI mostra progress `PDF M/N` come prima.

### Vantaggi

- **Non-blocking save**: utente preme GENERA, riceve risposta immediata
  dopo saveBatch (che e' gia' veloce, ~200-500ms). Compile gira in
  background, il browser polla lo status.
- **Retry automatico**: VPS down → job in retry, riprova fra 1m. Niente
  errore al primo fail.
- **Dedup**: stessa sha + variant + engine = stesso job. Click ripetuti
  GENERA non duplicano lavoro.
- **Multi-worker safe**: `pickNext()` atomic permette di scalare con
  multipli cron worker (`* * * * *` su 2-3 entry crontab).
- **Telemetry**: `countByStatus()` + worker summary JSON per monitoring.

### File toccati

- `database/migrations/033_verifica_compile_jobs.sql` (nuovo)
- `app/Repositories/VerificaCompileJobRepository.php` (nuovo)
- `app/Services/Verifica/VerificaCompileJobService.php` (nuovo)
- `app/Controllers/VerificaController.php` (compileAsync, getJob)
- `routes/web.php` (POST /compile-async, GET /jobs/{jobId})
- `tools/cron/process_compile_jobs.php` (nuovo)


### Compatibilita'

  synctex/aux).

## 2026-05-05 — Phase G22.S4.B.3 — VPS `/compile-bundle` endpoint multi-file

Slice S4.B.3 di 4. Il VPS `tex-compile-vps` ora ha un endpoint
**`/compile-bundle`** che riceve il bundle multi-file completo e compila
nativamente con `\input{...}` filesystem-based, eliminando il
flatten/inline-expansion lato Aruba.

### Problema risolto

Pre-S4.B.3: `compilePdf` doveva flatten() il bundle multi-file (storage
nativo S4.B.2) PRIMA di inviarlo al VPS via `/compile` legacy single-file.
Doppio lavoro: storage e' multi-file, ma il transit verso VPS era ancora
single-file. Inoltre il `.tex` flat e' meno leggibile in caso di errori
log (pacchetto inline invece che `\input`).

### Soluzione

**VPS Python (v1.2.0)**:

1. **`compile_bundle(files, main_path, ...)`** in `compile.py` —
   materializza i file in tmpdir isolato (validazione anti-traversal),
   lancia pdflatex/xelatex/lualatex sul `main_path`. Stessi flag
   anti-shell-escape, nonstopmode, semaphore concurrent del legacy.

2. **`POST /compile-bundle`** in `main.py` — pydantic
   `CompileBundleRequest` con array `files: [{path, content_b64}]` +
   `main_path` + `engine` + `passes`. Stessa auth HMAC del legacy
   `/compile`. Risposta identica (PDF binario o JSON con artefatti
   se `?with_artifacts=1`).

3. **Limiti**: 64 file max per richiesta, 10 MB totali bundle, 5 MB per
   singolo file via path validation.

**PHP client**:

4. **`TexCompileClient::compileBundle($files, $mainPath, $docId, ...)`** —
   wraps il POST `/compile-bundle` con HMAC. Stessa shape risposta del
   `compile()` legacy.

5. **`VerificaController::compilePdf`** — usa il bundle path quando:
   - Row e' multi-file (S4.B.2+): `tex_files` manifest popolata
   - **Nessun `tex_override`**: l'utente ha modificato il flat dal
     preview modal → fallback `/compile` single-file (no manifest del
     contenuto modificato disponibile)
   - Row legacy (pre-S4.B.2): fallback `/compile`

### Flusso bundle

```
[Browser GENERA]
  ↓ POST /api/verifica/save-tex-batch
[saveBatch: build(MODE_FLAT) → 8 BuildResult, ognuno cifrato file-by-file]
  ↓ tex_files JSON: [{path, blob_path, blob_kv, sha256}, ...] x N
[Browser compilePdfForDocs(docs)]
  ↓ POST /api/verifica/{id}/compile
[compilePdf:
   readManifestFiles(teacherId, id) → [{path, content}, ...]
   client.compileBundle(files, "versioni/main_NOR.tex", docId)
   ↓ HTTPS HMAC payload JSON con files base64-encoded
[VPS /compile-bundle:
   materialize tmpdir + pdflatex versioni/main_NOR.tex
   ↓ \input{texCommon/intestazione} risolve come ../texCommon/intestazione.tex
   ↓ \usepackage{../texCommon/verifica} carica verifica.sty come pacchetto
   ↓ PDF binario]
[Aruba attachPdf cifra envelope e salva pdf_blob_path]
```

## 2026-05-05 — Phase G22.S4.B.2 — Multi-file storage nativo verifica_documents

Slice S4.B.2 di 4. Il blob singolo monolitico (`tex_blob_path`) e' sostituito

Storage pre-S4.B.2: 1 row = 1 .tex monolitico. Costringeva il pipeline a

salvarli, e re-genrarare i bundle on-demand per ZIP/VSC download.

### Soluzione

1. **Migration `031_verifica_documents_tex_files.sql`**:
   - `ADD COLUMN tex_files LONGTEXT NULL`: manifest JSON con lista
     `[{path, blob_path, blob_kv, sha256, size}, ...]`

     restano popolate; nuove row hanno solo `tex_files`.

2. **VerificaDocumentRepository**:
   - `create()` accetta `tex_files` array (JSON encode automatico).
   - `texFiles(int $docId)` decodifica e ritorna lista manifest.
   - `hydrate()` decodifica `tex_files` JSON, normalizza
     `tex_blob_path/kv/size` come nullable.

3. **VerificaDocumentService**:

   - `readManifestFiles(teacherId, docId)`: ritorna lista `[{path, content}]`
     pronta per S4.B.3 `tar.gz` materialization verso VPS.
   - `deleteDoc`: cancella TUTTI i blob della manifest + legacy fallback.
   - `saveBatch` force-replace: raccoglie blob multi-file da reap.

4. **VerificaController + VerificaSyncService**:
   - `localBundle` accetta row sia multi-file sia legacy (non-skip).
   - Drive sync usa `readTex()` per ottenere flat per Drive upload (1 file

## 2026-05-05 — Phase G22.S4.B.1 — TexBuilder pipeline unification (`buildFlat`)

Slice S4.B.1 di 4 (FULL+migrate scope). Unifica il **build** pipeline: una

   usata per ZIP/VSC download. Templates da `storage/templates/verifiche/`,
   placeholder substitution, \input statements.
2. **`TexBuilder::buildLegacy($sel, $variant)` → `string` monolitica** —
   usata da `saveBatch` per il blob singolo. Preamble da
   `VersionPicker::baseHardcoded()` + body da `problems()` + `\end{document}`,
   con sezioni intestazione/griglia/criteri/footer iniettate via
   `VerificaDocumentService::applyTemplate()` post-process.

Risultato: il `.tex` cifrato in DB **NON era bit-identico** al `.tex`
distribuito in ZIP/VSC. Preamble, posizione `\maketitle`, modo di
includere stili — tutto leggermente diverso.

Inoltre 3 controller (`VerificaController::saveTex`,
`AdminPrintController::generate`, `TeacherPrintController::print`)
chiamavano `$tex->build()` aspettandosi `string` ma da G20.0 ritorna
`BuildResult` → endpoint **silenziosamente rotti** dal refactor.

### Soluzione

1. **`BuildResult::MODE_FLAT`** (rinominato durante review da
   `MODE_LEGACY`, troppo fuorviante): nuovo mode bundle multi-file con
   path canonici (`texCommon/...`, `versioni/...`, `griglie/...`).
2. **`BuildResult::flatten()`** — inline-espande `\usepackage{texCommon/X}`
   con il contenuto cleaned di `texCommon/X.sty` (strip
   `\NeedsTeXFormat`/`\ProvidesPackage`/`\endinput`) e ogni `\input{...}`
   con il contenuto del file referenziato. Risoluzione iterativa fino a
   5 livelli (defensive contro cicli).
3. **`TexBuilder::buildFlat($sel, $variant, $opts)`** — alias di
   `build($sel, $variant, ['mode' => MODE_FLAT])->flatten()`. Ritorna la
   stringa .tex monolitica direttamente. Sostituisce `buildLegacy`.
4. **`saveBatch`/`saveTex` drop di `applyTemplate`** — i template
   `main_*.tex` referenziano gia' intestazione/griglia/ulteriori_misure/
   BES_DSA via `\input`, quindi `flatten()` produce il `.tex` con tutte
   le sezioni inline. Niente piu' string-substitution post-build.
5. **Bonus fix**: 3 controller broken aggiornati a `buildFlat()`.
6. **Cleanup TexBuilder**: dead code rimosso (`problems()` privato,
   `Formatter`/`TableRenderer`/`Sanitizer` use unused).

## 2026-05-05 — Phase G22.S2 — Cache PDF content-addressed (SHA256 per-teacher)

Slice S2 di 6 — riduce le compile ridondanti su VPS tex-compile-vps.

### Problema risolto

Ogni `POST /api/verifica/{id}/compile` invocava sempre il VPS pdflatex
anche quando il sorgente .tex era identico a una compile precedente
(stesso docente, stessa selezione). Esempi tipici:

- Risalvi una verifica senza modifiche → 8 PDF identici ricompilati.
- Modifichi solo il titolo → 7 su 8 hanno il body identico.
- Varianti A_NOR e B_NOR a volte coincidono (stessi esercizi).

Costo: 3-5s pdflatex × 8 varianti × N risalvi/giorno = CPU VPS sprecata
+ latenza utente non necessaria.

### Soluzione — cache content-addressed

1. **Migration `030_verifica_documents_tex_sha256.sql`**:
   - `ALTER TABLE verifica_documents ADD COLUMN tex_sha256 CHAR(64) NULL`
   - `CREATE INDEX idx_verifica_docs_teacher_sha (teacher_id, tex_sha256)`
   - Nullable per compat con row legacy (pre-S2).

2. **Persistenza sha**:
   - `saveTex` / `saveBatch`: `hash('sha256', $tex)` salvato per ogni nuova row.
   - `updateTex` (preview modal `save_tex=true`): aggiorna sha al cambio TEX.

3. **`VerificaDocumentRepository::findCachedPdf(teacherId, sha256, excludeId)`**:
   query indexata: cerca un altro doc dello stesso docente con stesso
   sha + `pdf_blob_path IS NOT NULL`. ORDER BY pdf_uploaded_at DESC LIMIT 1.

4. **`VerificaDocumentService::attachCachedPdfFor(teacherId, docId, texSource)`**:
   - calcola sha del texSource
   - findCachedPdf, se hit:
     1. decrypt PDF cached con TKEK del docente (envelope ADR-006)
     2. valida magic bytes `%PDF-` (defensive)
     3. attachPdf alla row corrente (cifra envelope nuovo blob)
   - graceful fallback: eccezioni decrypt / magic invalido → `null` (cache miss)

5. **`VerificaController::compilePdf` integrazione**:
   ```
   $texSource = override || readTex
   if (!withArtifacts) {
     $cached = svc->attachCachedPdfFor(teacherId, id, texSource)
     if (cached !== null) {
       return ['ok'=>true, 'compile'=>['engine'=>'cache', 'duration_ms'=>0, 'cache_hit'=>true]]
     }
   }
   // existing flow: VPS compile
   ```
   Cache disabilitata se `?with_artifacts=1` (preview modal richiede synctex_gz/aux/log non in cache).

### Sicurezza — scope per-teacher OBBLIGATO

Cross-teacher cache violerebbe l'envelope encryption ADR-006: per
decifrare un PDF di un altro docente servirebbero 2 TKEK in chiaro
contemporaneamente lato server (privilege escalation surface).

Per-teacher: il docente proprietario detiene la propria TKEK → puo'
decifrare i PROPRI PDF cached e ri-cifrarli per nuove row. Niente leak
fra docenti.

### File toccati

- `database/migrations/030_verifica_documents_tex_sha256.sql` (nuovo)
- `app/Repositories/VerificaDocumentRepository.php` (create+findCachedPdf+updateTexBlob+sha)
- `app/Services/Verifica/VerificaDocumentService.php` (saveTex/saveBatch/updateTex sha + attachCachedPdfFor)
- `app/Controllers/VerificaController.php` (compilePdf cache check)
- `tests/Unit/Services/Verifica/VerificaDocumentServiceCacheTest.php` (nuovo, 6 test)

### Test

```bash

# 6 test, tutti pass: saveTex_persists_sha256, miss, hit, magic invalido,
# decrypt exception, tex source vuoto.
```

### Trade-off

- **Storage cresce**: ogni hit duplica il blob PDF cifrato (envelope DEK
  diverso per la nuova row). PDF tipici 200KB-2MB → impatto modesto.
  Niente refcount cross-row (deliberato: ownership per-row evita orfani).
- **Hit-rate atteso 40-60%**: utente risalva spesso; varianti con stesso

- **Override non in cache**: `with_artifacts=1` (preview modal) bypassa
  cache perche' richiede synctex_gz binario. Fix futuro: serializzare

### Rollback

Disabilitazione runtime: niente switch — la cache e' implicita nella
query. Per disabilitarla: rimuovere o commentare il blocco `if
(!$withArtifacts)` in `VerificaController::compilePdf`. Migration
reversibile via `DROP INDEX + DROP COLUMN`.

## 2026-05-05 — Phase G22.S1 — saveBatch / saveTex atomicità + cleanup blob orfani

Hardening della pipeline di produzione verifiche (slice S1 di 6).

### Problema risolto

`VerificaDocumentService::saveBatch` creava fino a 8 row + 8 blob cifrati
senza transazione DB. Se la N-esima `INSERT` falliva: N−1 row in DB +
N−1 blob orfani su `storage/verifiche_enc/`. Stesso pattern in `saveTex`
(1 blob + 1 row non-atomic). Inoltre il branch `force=1` chiamava
`store->delete($teacherId, $relPath)` (firma errata: 2 arg invece di 1)
→ `guardPath` falliva silente → blob force-replaced rimanevano sul disco.

### Soluzione

1. **TransactionRunner astratto** (`app/Support/TransactionRunner.php`)
   con default `PdoTransactionRunner` su `Database::connection()`. Permette
   ai servizi di dichiarare boundary transazionali senza PDO statico
   diretto, e di sostituire un `NullTransactionRunner` nei test.
2. **saveTex**: `store.put` resta fuori dalla tx (atomic via tmp+rename),
   `docs.create` dentro `tx->run`. Su throw → cleanup del blob appena scritto.
3. **saveBatch** ristrutturato in 3 fasi:
   - **Phase 1** (no-tx): build+template+`store.put` per N varianti, blob
     tracciati in `$writtenBlobs`.
   - **Phase 2** (tx): delete row esistenti (force) + insert N nuove. Tutte
     o nessuna. Su rollback → cleanup `$writtenBlobs`.
   - **Phase 3** (post-commit): reap dei blob force-replaced raccolti in
     `$blobsToReap`. Best-effort: eventuale fallimento lascia file orfani
     su disco ma DB consistente.
4. **`safeDeleteBlob`/`safeDeleteBlobs`** helper: `exists()` + `delete()`
   con try-swallow per non mascherare l'eccezione originale del rollback.
5. **Bug fix** `force=1`: chiamata `store->delete` ora con singolo arg.
6. **Testabilità**: `VerificaDocumentRepository`, `VerificaTemplateRepository`,
   `EncryptedBlobStore` non sono piu' `final` (mockabili via subclass).

- `app/Support/TransactionRunner.php` (nuovo, interface)
